# Аналітична панель "LevelUp" - Повний технічний огляд

Цей документ надає поглиблений опис архітектури, функціональних модулів та технічних особливостей веб-додатку "LevelUp Analytics".

## 1. Основна концепція та технології

**Призначення:** Інтерактивна аналітична панель для обробки та візуалізації даних про продуктивність роздрібних точок (ритейл) та аналітики продажів. Додаток розроблено для роботи "на льоту" безпосередньо в браузері, без необхідності у серверній інфраструктурі.

**Ключові технології:**
- **Frontend Framework:** Next.js (з App Router), React, TypeScript.
- **UI-компоненти:** `shadcn/ui` для базових елементів інтерфейсу (кнопки, діалоги, таблиці тощо).
- **Стилізація:** Tailwind CSS для стилів, з кастомною темою в `src/app/globals.css`.
- **Обробка Excel:** Бібліотека `xlsx` (`sheetjs`) для парсингу `.xlsx` та `.xls` файлів.
- **Зберігання даних:** `localStorage` браузера використовується як основне сховище для кешування оброблених даних, звітів та налаштувань користувача, що забезпечує збереження стану між сесіями.

## 2. Архітектура додатку

Додаток побудований за модульним принципом. Кожен з трьох основних модулів функціонує переважно незалежно, зберігаючи власні дані та налаштування у `localStorage`.

### Головний компонент (`src/app/page.tsx`)
Це центральний вузол додатку, який відповідає за:
- **Навігацію** між основними екранами (`main-menu`, `statistics`, `sales`, `sms-form`).
- **Ініціалізацію та завантаження даних** з `localStorage` при старті.
- **Обробку завантаження файлів** для модуля статистики ритейлу.
- **Управління глобальним станом** (поточний вид, завантажені дані, дата імпорту).

### Зберігання даних (`localStorage`)
- `analyticsData`: Кешує всі дані, пов'язані з модулем **статистики ритейлу** (вихідні дані, оброблені звіти, групи аутлетів, дата).
- `salesData`, `salesColumnMapping`, `salesCodifierTemplate`: Кешують дані, налаштування парсера та шаблон кодифікатора для модуля **аналітики продажів**.
- `orderData`, `orderColumnMapping`: Кешують дані та налаштування для **бланка замовлення**.

## 3. Модуль 1: Статистика ритейлу (`statistics-dashboard`)

**Призначення:** Аналіз звітів про виконання планових показників (факт/рівні) торговими точками.

### Логіка роботи:

**1. Імпорт даних (`handleFileUpload` в `page.tsx`):**
- Користувач може завантажити один або кілька файлів `.xlsx`.
- Система автоматично визначає тип аркуша:
    - **Основний звіт:** Береться перший аркуш файлу.
    - **PrP/PoP звіти:** Шукаються аркуші, назви яких містять ключові слова `PrP` або `PoP` (за допомогою регулярних виразів).

**2. Обробка та агрегація даних (Ключова особливість):**
- **Об'єднання рядків:** Критично важливий етап. Часто дані для одного аутлету розбиті на кілька рядків у вихідному файлі. Система агрегує **всі рядки**, що стосуються одного унікального аутлету, в єдиний об'єкт. Це гарантує, що для аналізу береться повна інформація (всі факти та всі рівні).
- **Динамічний пошук стовпців:** Замість прив'язки до фіксованих назв (`A`, `B`, `C`), функція `findRelatedColumns` шукає потрібні стовпці за ключовими словами в їх заголовках (напр., "Активації PrP+PoP", "Факт", "Рівень1"). Це робить систему стійкою до змін у структурі файлів.

**3. Візуалізація (Звіти та дашборди):**
- **Vf-View (`vf-view.tsx`):** Основний дашборд з картками показників. Колір картки (`glass-red`, `achieved`, `glass-yellow`, `glass-green`) динамічно визначається на основі найвищого досягнутого рівня.
- **Flexbox Report (`flexbox-report.tsx`):** Компактний звіт, що відображає всі аутлети та їхні ключові показники у вигляді невеликих карток на одній сторінці. Зручний для швидкого огляду.
- **PrP/PoP Reports (`prp-report.tsx`, `pop-report.tsx`):** Спеціалізовані табличні звіти, що генеруються на основі відповідних аркушів Excel. Надають детальну інформацію з можливістю фільтрації.
- **Аналіз темпів (`pacing-dialog.tsx`):** Розраховує необхідний щоденний темп для досягнення цільових показників до кінця місяця.

## 4. Модуль 2: Аналітика продажів (`sales-analytics`)

**Призначення:** Поглиблений аналіз детальних звітів про продажі з можливістю гнучкого налаштування та класифікації даних.

### Логіка роботи:

**1. Імпорт та налаштування парсера:**
- Користувач завантажує один файл зі звітом про продажі.
- **Парсер (`ColumnMapping`):** Користувач **один раз** має вказати, які стовпці у файлі відповідають за ключові сутності: "Опис" (співробітник), "Код товару", "Назва", "Вартість", "Відділ" та "IMEI".
- Ці налаштування зберігаються в `localStorage` і застосовуються автоматично при наступних завантаженнях файлів зі схожою структурою.

**2. Кодифікатор (`CodifierManager`):**
- Це потужний інструмент для очищення та уніфікації "сирих" даних. Він працює в кілька етапів, і кожен етап зберігає свій шаблон у `localStorage`.
- **Опис / Відділ:** Дозволяє згрупувати різні варіанти написання (напр., "Іванов І. І.", "Ivanov I.") в одне чисте ім'я ("Іванов").
- **Код / Смартфони (режим ключових слів):** На відміну від попереднього, цей режим працює за **ключами**. Користувач вводить ключове слово (напр., "case"), і всі товари, у назві яких є це слово, отримують єдиний уніфікований код ("Чохол"). Розділення на "Код" (товари без IMEI) та "Смартфони" (товари з IMEI) дозволяє уникнути конфліктів класифікації.
- **Індекс:** Фінальний етап класифікації. Дозволяє призначити групу (індекс) для кожного уніфікованого коду товару (напр., код "iPhone 15 Pro" потрапляє в індекс "Елітні смарти", а "Чохол" — в "Акси").

**3. Аналіз (`AnalyticsView`):**
- Після налаштування парсера та кодифікатора, користувач отримує доступ до екрана аналітики.
- Дані групуються за індексами або співробітниками, надаючи зведену інформацію про кількість та суму продажів.
- Передбачена гнучка фільтрація та сортування.

## 5. Модуль 3: Форма для SMS (`sms-form`)

**Призначення:** Швидке та безпомилкове генерування SMS-запитів на сервісні номери 150 та 640.

**Логіка роботи:**
- Користувач обирає тип сервісу.
- Вводить номер телефону та (для сервісу 150) код.
- Форма валідує введені дані в реальному часі (формат номера, діапазон коду).
- Після підтвердження, програма генерує `sms:` посилання і передає його операційній системі, яка відкриває стандартний додаток для надсилання SMS з уже заповненим номером отримувача та текстом повідомлення.
- Це повністю клієнтське рішення, яке не надсилає SMS самостійно, а лише готує їх.

## 6. Візуальний дизайн та UI/UX

**Основна ідея:** Створення сучасного, "преміального" інтерфейсу з ефектом "скляної" прозорості (Glassmorphism), темною темою та яскравими акцентами, що асоціюються з брендом.

### 6.1. Колірна палітра та фон

- **Фон:** Динамічний радіальний градієнт, визначений у `src/app/globals.css`. Основний фон — від темно-синього (`#0a1a3a`) до майже чорного (`#020408`), що створює глибину. Для екрана SMS фон змінюється на червоний градієнт (`#5e0000`).
- **Основні кольори:** Визначені як CSS-змінні для легкого керування темою.
  - `--vodafone-red: #e60000`: Фірмовий червоний колір для активних елементів.
  - `--glass-hub-active`, `--glass-hub-muted`: Сині відтінки для інтерактивних елементів у стилі "Conn-Hub".
  - `--glass-red-muted`, `--glass-yellow-muted`, `--glass-green-muted`: Кольори для позначення статусу виконання планів (червоний, жовтий, зелений) з ефектом скла.

### 6.2. Стиль "Glassmorphism" (Скляний ефект)

Це ключовий елемент дизайну, який застосовується до більшості інтерактивних блоків (карток, кнопок, діалогів).

- **Клас `.glass-card`:**
  - `backdrop-filter: blur(50px) saturate(1.8)`: Створює ефект розмитого фону під елементом.
  - `background: linear-gradient(...)`: Напівпрозорий лінійний градієнт.
  - `border: 1.5px solid var(--border-white)`: Тонка біла рамка для імітації краю скла.
  - `box-shadow`: Створює тінь, що "підіймає" елемент над фоном.

- **Кастомні стилі для кнопок та карток:**
  - **`.conn-hub-style`**: Варіація "скляної" картки в синіх тонах для навігаційних елементів.
  - **`.sms-btn`, `.glass-btn-blue`**: Спеціалізовані стилі для кнопок, які також використовують ефект скла.
  - **`.achieved`**: Спеціальний клас, який змінює вигляд картки/кнопки при досягненні мети (робить її яскравішою, змінює тінь).

### 6.3. Типографіка та верстка

- **Шрифт:** Використовується `Inter` з Google Fonts для всього тексту, що забезпечує відмінну читабельність.
- **Верстка:** Побудована на `Flexbox` та `Grid` з Tailwind CSS, що робить інтерфейс повністю адаптивним.
- **Відступи:** Використовується система відступів Tailwind (`p-4`, `gap-4` тощо) для створення чистої та послідовної структури.

### 6.4. Адаптивність (Mobile-First)

- **Нижня навігація (`BottomNavBar`):** На мобільних пристроях з'являється фіксована нижня панель для зручної навігації однією рукою. На десктопі вона прихована.
- **Приховування елементів:** Деякі елементи (наприклад, кнопка "Назад" на головному меню) приховані на мобільних пристроях, щоб не захаращувати інтерфейс.
- **Динамічні відступи:** Головний контейнер має динамічний нижній відступ (`pb-[120px]` на мобільному), щоб контент не перекривався нижньою навігаційною панеллю.
